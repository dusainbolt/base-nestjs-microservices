# RabbitMQ Reliability — Khi nào down và cách phòng tránh

*Ngày tạo: 03-04-2026*

---

## 1. Nguyên nhân RabbitMQ down

### 1.1. Hết RAM (Phổ biến nhất)
RabbitMQ giữ message trong memory. Khi RAM đạt ngưỡng mặc định (40% tổng RAM), nó sẽ **block tất cả producer** — không nhận message nào nữa.

```
Dấu hiệu : Producer timeout, connection bị treo
Ngưỡng   : vm_memory_high_watermark = 0.4 (40% RAM)
Ví dụ    : Server 4GB RAM → block khi RabbitMQ dùng > 1.6GB
```

### 1.2. Hết Disk
Persistent messages được ghi xuống disk. Khi disk free còn dưới ngưỡng (mặc định 50MB) → block toàn bộ hệ thống.

### 1.3. Queue phình to (Consumer chậm hơn Producer)
Khi tốc độ publish >> tốc độ consume, queue tích tụ message → RAM tăng dần → cuối cùng trigger watermark → block.

```
Ví dụ thực tế:
  Producer: 1000 msg/s
  Consumer: 500 msg/s
  → Mỗi giây tồn đọng 500 msg
  → Sau 1 giờ: 1,800,000 msg chờ xử lý
  → RAM có thể tăng hàng GB
```

### 1.4. Quá nhiều Connection/Channel
Mỗi microservice mở connection + channel. Nếu không đóng đúng cách (leak), RabbitMQ sẽ cạn file descriptor của OS.

### 1.5. Network Partition (Cluster)
Trong RabbitMQ cluster (≥ 2 nodes), nếu 2 node mất kết nối → split-brain → mỗi node tự coi mình là "primary" → có thể mất hoặc duplicate message.

### 1.6. Maintenance / Human Error
Restart, upgrade version, hoặc thay đổi cấu hình sai (ví dụ: xóa nhầm exchange/queue).

## 2. Các metric quan trọng cần theo dõi

| Metric | Ngưỡng cảnh báo | Ý nghĩa |
| :--- | :--- | :--- |
| **Memory used** | > 70% watermark | Sắp bị block producer |
| **Disk free** | < 200MB | Sắp bị block toàn bộ |
| **Queue depth** (messages ready) | > 10,000 | Consumer không kịp xử lý |
| **Unacked messages** | > 1,000 | Consumer bị stuck hoặc chậm |
| **Connection count** | Tăng bất thường | Có thể leak connection |
| **Publish rate vs Consume rate** | Publish >> Consume | Queue sẽ phình to dần |
| **Channel count** | > 500 | Quá nhiều channel mở đồng thời |

## 3. Công cụ giám sát

### 3.1. RabbitMQ Management UI (có sẵn)
```
URL: http://localhost:15672
User/Pass: guest/guest (dev) hoặc admin credentials (prod)
```
Cung cấp biểu đồ realtime cho: connections, channels, queues, message rates, memory và disk usage.

### 3.2. Prometheus + Grafana (Production)
RabbitMQ có plugin `rabbitmq_prometheus` tích hợp sẵn:
```bash
# Bật plugin
rabbitmq-plugins enable rabbitmq_prometheus

# Metrics endpoint
GET http://localhost:15692/metrics
```

Kết hợp với Grafana dashboard (ID: `10991`) để có biểu đồ chuyên nghiệp và alert tự động khi chạm ngưỡng.

### 3.3. Health Check API
```bash
# Check tổng thể
GET http://localhost:15672/api/healthchecks/node

# Check từng thành phần
GET http://localhost:15672/api/aliveness-test/%2F
```

## 4. Biện pháp phòng tránh

### 4.1. Đã triển khai trong hệ thống ✅

| Biện pháp | Mô tả |
| :--- | :--- |
| **Durable queues** | Queue không bị mất khi RabbitMQ restart |
| **Persistent messages** | Message được ghi xuống disk |
| **Dead Letter Queue (DLQ)** | Lỗi 5xx được chuyển sang DLQ thay vì mất |
| **ACK/NACK** trong `RmqInterceptor` | Message chỉ bị xóa khi xử lý thành công |
| **Connection retry** | Auto reconnect khi mất kết nối tạm thời |

### 4.2. Nên triển khai thêm 🔜

| Biện pháp | Mô tả | Độ ưu tiên |
| :--- | :--- | :--- |
| **Queue max-length** | Giới hạn số message tối đa trong queue, tránh phình RAM | Cao |
| **TTL per message** | Message tự hủy sau N giây nếu không được consume | Trung bình |
| **RabbitMQ cluster (≥ 2 nodes)** | High Availability, tự failover khi 1 node chết | Cao (prod) |
| **Monitoring + Alert** | Prometheus/Grafana alert khi RAM/disk chạm ngưỡng | Cao (prod) |
| **Lazy queues** | Ghi message xuống disk ngay thay vì giữ RAM, phù hợp queue lớn | Thấp |
| **Consumer prefetch** | Giới hạn số message consumer nhận cùng lúc, tránh overload | Trung bình |

## 6. Kết luận

> **Rủi ro thật sự không phải "RabbitMQ chết"**, mà là **queue phình to do consumer chậm** → hết RAM → block producer. Đây là thứ **có thể monitor và phòng tránh hoàn toàn** bằng alerting.

Với cấu hình hiện tại (durable + persistent + DLQ + ACK/NACK), hệ thống đã an toàn cho giai đoạn phát triển. Khi lên production, ưu tiên triển khai **monitoring** và **cluster** trước khi cân nhắc Outbox Pattern.
