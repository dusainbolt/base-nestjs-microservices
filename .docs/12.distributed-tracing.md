# Distributed Tracing — Theo dõi vết với OpenTelemetry & Jaeger

*Ngày triển khai: 03-04-2026*

---

## 1. Giới thiệu

Trong hệ thống Microservices, một request từ Client có thể đi qua Gateway, gọi RPC sang nhiều service khác, và bắn ra các Domain Event. **Distributed Tracing** giúp chúng ta nhìn thấy toàn bộ "hành trình" này trên một trục thời gian duy nhất để:
- Tìm điểm nghẽn (Bottlenecks)
- Debug các request bị lỗi chéo (Cascade failures)
- Hiểu rõ sự tương quan giữa các service.

Hệ thống sử dụng **OpenTelemetry (OTEL)** để thu thập dữ liệu và **Jaeger** để hiển thị.

---

## 2. Cách khởi chạy Jaeger Infrastructure

Dùng Docker Compose để chạy Jaeger:

```bash
docker compose -f .compose/jaeger/docker-compose.yml up -d
```

Sau đó, truy cập Jaeger UI tại: **[http://localhost:16686](http://localhost:16686)**

## 3. Cách thức hoạt động trong Code

### 3.1. Khởi tạo Tracing (SDK)
Mọi microservice đều được tích hợp `initTracing` từ `@app/common`. Điều quan trọng nhất là `initTracing` phải được gọi **ngay đầu file `main.ts`**, trước tất cả các import khác để OpenTelemetry có thể can thiệp (monkey-patch) vào các thư viện như `http`, `express`, `amqplib`.

**File `main.ts` mẫu:**
```typescript
import { initTracing } from '@app/common';
initTracing('product-service'); // Tên service hiển thị trên Jaeger

import { NestFactory } from '@nestjs/core';
// ... các import khác ...
```

### 3.2. Chế độ theo dõi (Instrumentation)
Hệ thống tự động theo dõi các thành phần sau:
- **HTTP/Express**: Tự động tạo span cho mọi request incoming/outgoing.
- **NestJS Core**: Theo dõi các controller, provider internal spans.
- **AMQPLib (RabbitMQ)**: **Quan trọng nhất.** Tự động gắn `traceparent` vào message header khi gửi/nhận, giúp nối liền vết xuyên qua RabbitMQ.
- **Database (Postgres)**: Tự động bắt mọi câu lệnh SQL (SELECT, INSERT, UPDATE, DELETE) từ Prisma qua Postgres.
- **Redis (IORedis)**: Theo dõi mọi lệnh `GET`, `SET`, `MGET`, v.v. gửi tới Redis.

## 4. Bạn có thể xem được gì trong một Trace?

Ngoài thời lượng, Jaeger cho phép bạn đào sâu vào dữ liệu chi tiết của từng bước:

### 4.1. Xem SQL Query
Khi bạn click vào một span của Postgres, trong tab **Tags/Attributes**, bạn sẽ thấy:
- `db.statement`: Câu lệnh SQL thực tế vừa chạy.
- `db.name`: Tên database.
- `db.user`: User đang thực thi.

### 4.2. Xem Redis Commands
Các span của Redis sẽ cho bạn thấy:
- Tên lệnh (GET, SET...).
- Key đang thao tác (một số phiên bản sẽ ẩn payload để bảo mật nhưng vẫn thấy được lệnh).

### 4.3. Lưu Payload & Metadata (Attributes)
Bạn có thể gắn thêm dữ liệu tùy chỉnh vào mã nguồn để lưu vết, ví dụ:
```typescript
import { trace } from '@opentelemetry/api';

const span = trace.getActiveSpan();
if (span) {
  span.setAttribute('user.id', '123');
  span.setAttribute('payload', JSON.stringify(data));
}
```
*Lưu ý: Không nên lưu các dữ liệu nhạy cảm như Password hay Credit Card vào Trace.*

### 4.4. Theo dõi lỗi (Events & Exceptions)
Khi một service ném ra Exception, OpenTelemetry tự động bắt được:
- Message lỗi.
- **Stack trace** chi tiết (giúp biết chính xác lỗi ở dòng nào mà không cần mở log file).

## 5. Cách theo dõi một Request

Giả sử bạn thực hiện `GET /products?include=createdBy`:

1. **API Gateway** nhận request → Bắt đầu một `Trace ID`.
2. **Gateway** gọi RPC sang **Product Service** → `Trace ID` được truyền qua RabbitMQ header.
3. **Product Service** nhận lệnh → Tiếp tục `Trace ID` đó, tạo span mới.
4. **Gateway** gọi RPC sang **User Service** (để enrich) → Tiếp tục truyền `Trace ID`.
5. **Jaeger** sẽ hiển thị một sơ đồ cây cho thấy request này mất tổng cộng bao nhiêu lâu và mỗi service con chiếm bao nhiêu giây.

## 6. Lưu ý khi Phát triển

- **Trace Propagation**: Khi bạn gọi một service khác qua HTTP hoặc RabbitMQ bằng các công cụ chuẩn (Axios, ClientProxy), OTEL sẽ tự động lo việc truyền ID. Nếu bạn dùng các thư viện tự chế không được patch, trace sẽ bị đứt.
- **Performance**: Việc tracing có tốn tài nguyên? Có, nhưng rất ít. Hệ thống sử dụng `BatchSpanProcessor` để gửi dữ liệu theo lô thay vì gửi từng request một, giúp giảm thiểu overhead.
- **Graceful Shutdown**: SDK được cấu hình để flush toàn bộ spans còn dư xuống Jaeger trước khi ứng dụng tắt hẳn.

## 7. Giao diện Jaeger UI

- **Service**: Chọn service cần xem (e.g. `api-gateway`).
- **Operation**: Chọn endpoint (e.g. `GET /products`).
- **Timeline View**: Xem biểu đồ Gantt của request.
- **Trace Graph**: Xem sơ đồ quan hệ giữa các component.
