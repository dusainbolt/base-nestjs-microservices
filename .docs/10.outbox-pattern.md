# Outbox Pattern — Đảm bảo tính nhất quán khi gửi Domain Event

*Ngày tạo: 03-04-2026*
*Trạng thái: Tài liệu tham khảo — Chưa triển khai*

---

## 1. Vấn đề cần giải quyết

Trong kiến trúc Microservices, khi một service thực hiện **ghi DB** và **gửi Event** (qua RabbitMQ), hai thao tác này không nằm trong cùng một transaction. Điều này dẫn đến rủi ro mất nhất quán dữ liệu.

### Ví dụ thực tế: Luồng xóa User

**Kịch bản 1 — RabbitMQ down:**
```
Auth Service:
  1. Xóa user trong DB              ✅ Thành công
  2. Emit event USER_DELETED         ❌ RabbitMQ đang down!
     → Product Service KHÔNG nhận được event
     → Product của user đó KHÔNG BAO GIỜ bị cleanup
     → Dữ liệu "mồ côi" tồn tại mãi mãi
```

**Kịch bản 2 — DB rollback sau khi đã gửi event:**
```
Auth Service:
  1. Emit event USER_DELETED         ✅ Gửi thành công
  2. Xóa user trong DB              ❌ DB lỗi, rollback!
     → Product Service đã nhận event và xóa product
     → Nhưng User vẫn còn tồn tại!
     → Dữ liệu bị mất oan
```

**Nguyên nhân gốc:** Hai thao tác (ghi DB + gửi message) là hai hệ thống riêng biệt, không thể gói chung trong một ACID transaction.

## 2. Giải pháp: Outbox Pattern

### Ý tưởng cốt lõi

Thay vì gửi event trực tiếp vào RabbitMQ, **ghi event vào một bảng `outbox` trong cùng DB và cùng transaction** với thao tác nghiệp vụ. Một background worker sẽ đọc bảng này và gửi event đi.

### Luồng hoạt động

```
┌─────────────────────────────────────────────────────┐
│  Auth Service                                       │
│                                                     │
│  BEGIN TRANSACTION                                  │
│    1. DELETE FROM users WHERE id = 'user-123'       │
│    2. INSERT INTO outbox (event_type, payload, ...)  │
│       VALUES ('USER_DELETED', '{"userId":"..."}')   │
│  COMMIT                                             │
│  → Cả 2 đều thành công HOẶC đều rollback           │
└─────────────────────┬───────────────────────────────┘
                      │
          (Background Worker / Cron Job)
                      │
                      ▼
┌─────────────────────────────────────────────────────┐
│  Outbox Publisher                                   │
│                                                     │
│  1. SELECT * FROM outbox WHERE status = 'PENDING'   │
│  2. Publish từng event vào RabbitMQ                 │
│  3. UPDATE outbox SET status = 'SENT'               │
│     (hoặc DELETE nếu không cần audit trail)         │
└─────────────────────────────────────────────────────┘
```

### Schema bảng Outbox (Prisma)

```prisma
model Outbox {
  id          String   @id @default(uuid())
  eventType   String   // USER_DELETED, ORDER_CREATED, ...
  payload     Json     // Nội dung event
  status      String   @default("PENDING") // PENDING | SENT | FAILED
  retryCount  Int      @default(0)
  createdAt   DateTime @default(now())
  processedAt DateTime?
}
```

## 3. So sánh với cách làm hiện tại

| Tiêu chí | Cách hiện tại (Direct Publish) | Outbox Pattern |
| :--- | :--- | :--- |
| **Ghi DB + Gửi Event** | 2 bước tách rời, không đảm bảo | 1 transaction duy nhất, atomic |
| **RabbitMQ down** | Event mất vĩnh viễn | Event an toàn trong DB, gửi lại khi phục hồi |
| **DB rollback** | Event đã gửi, không thu hồi | Event chưa commit → không tồn tại → không gửi |
| **Độ phức tạp** | Đơn giản | Cần bảng outbox + worker + retry logic |
| **Hiệu năng** | Nhanh hơn (ít bước) | Chậm hơn một chút (thêm polling interval) |
| **Phù hợp** | Prototype, MVP, low-risk data | Production, dữ liệu quan trọng |

---

## 4. Ưu điểm

1. **Đảm bảo At-Least-Once Delivery**: Event chắc chắn được gửi ít nhất một lần (worker sẽ retry nếu thất bại).
2. **Atomic**: Nghiệp vụ và event được ghi trong cùng một transaction — không bao giờ bị lệch.
3. **Audit Trail**: Bảng outbox là lịch sử tự nhiên của mọi domain event, hữu ích cho debugging.
4. **Resilience**: Hệ thống vẫn hoạt động bình thường khi RabbitMQ tạm thời down.

## 5. Nhược điểm

1. **Tăng độ phức tạp**: Cần thêm bảng DB, background job, và logic xử lý trùng lặp (idempotency).
2. **Độ trễ (Latency)**: Event không được gửi ngay lập tức mà phụ thuộc vào polling interval của worker (thường 1–5 giây).
3. **Idempotency bắt buộc**: Consumer phải xử lý được event trùng lặp vì worker có thể gửi lại event đã gửi (trong trường hợp crash giữa chừng).
4. **DB load**: Polling liên tục bảng outbox có thể tạo thêm tải cho database.

---

## 6. Khi nào nên triển khai?

### ✅ NÊN dùng khi:
- Hệ thống production xử lý dữ liệu quan trọng (tài chính, đơn hàng, thanh toán)
- Có nhiều Domain Event phức tạp chéo giữa các service
- Yêu cầu **zero data loss** và audit trail

### ❌ CHƯA CẦN khi:
- Dự án đang ở giai đoạn MVP, học tập, hoặc base project
- RabbitMQ đã được cấu hình đúng (durable queues, persistent messages)
- Mức độ chấp nhận rủi ro mất event thấp là chấp nhận được

---

## 7. Đánh giá cho hệ thống hiện tại

Hệ thống hiện tại đã có các biện pháp bảo vệ tốt:
- **Durable queues** + **Persistent messages** trên RabbitMQ
- **Dead Letter Queue (DLQ)** cho các lỗi 5xx
- **ACK/NACK** xử lý đúng trong `RmqInterceptor`

→ **Kết luận**: Outbox Pattern là **roadmap dài hạn**, nên triển khai khi hệ thống scale lên production thật với dữ liệu nhạy cảm. Hiện tại, cơ chế hiện có đã đủ an toàn cho giai đoạn phát triển.
